# daily_log_rules.md

## STRICT EXECUTION CONTRACT

本契约位于本规则文件之首，优先级高于本文件所有其他章节。

1. 本规则文件中所有标注“必须”“禁止”“仅”“只”“固定”的条款，执行优先级最高，不允许模型自行改写、合并、跳过或自由发挥。

2. 用户说「日报」后，必须进入当天日报记录模式，并读取 source config、HealthData 中前一天和当天的全部数据、当天「搬砖手册」日历事件、过去日报事件 event_id，写入本 chat 缓存。

3. 用户说「评估」后，必须基于本 chat 当天缓存与用户当天输入生成最终记录。

4. 最终输出必须且仅包含 10 块内容，顺序固定：
   1. event_id 行
   2. 标题
   3. 【💼 工作】
   4. 【📚 学习/数据】
   5. 【🏠 生活】
   6. 【🌿 游玩/体验】
   7. 【🧠 感想/感悟】
   8. 【⚠️ 卡点/问题】
   9. 【➡️ 明天接着干】
   10. 【📊 今日评估】

5. 禁止新增栏目、删除栏目、改栏目名、改顺序。

6. 禁止更新 Google Calendar。最终只输出可复制文本。

7. 禁止创建日历事件。

8. 禁止补写用户没说过的时间、地点、任务、结果、原因。

9. 今日评估只能基于当天缓存与用户输入，不允许使用外部信息或主观脑补。

10. 生成最终输出前，必须执行第 11 节输出前自检；任一项不通过必须重写。

## 0. 任务身份

任务名：日报。

记录对象：从用户说「日报」后，在同一 chat 内输入的当天事项。

数据依据：
- 用户当天输入内容
- HealthData 中前一天和当天的全部数据
- 当天「搬砖手册」日历事件
- 过去日报事件 event_id

输出对象：一份固定结构的「每日生活 M/D」，覆盖工作、学习、生活、游玩、体验、感想与次日计划。

## 1. Config 来源

唯一 Config 来源为本 Project 的 source 文件夹中的 config 文件，获取方式与 sleep_rules.md 完全一致。

执行「日报」任务前，必须先读取 source config。

必须从 source config 获取：
- Google Calendar 区块下「搬砖手册」的 calendar_id，用于读取当天日历事件与日报事件 event_id。
- Google Sheets 区块下 HealthData 的 Google Sheet ID，用于读取 report_date 前一天和 report_date 当天的全部健康数据。

禁止从外部 Google Doc 读取配置。
禁止从旧文件、缓存文件、PROJECT_CONFIG.md 或其他 md 文件替代 source config。
禁止写死 calendar_id、spreadsheet_id 或 sheet/tab 名。
禁止从用户输入中猜测数据源。

## 2. 触发词

- 「日报」：开始记录当天事项，并执行数据读取与缓存。
- 「评估」：生成当天每日生活记录，并用 1-2 句话概括这一天。

## 3. report_date 定义

report_date 默认为用户触发「日报」当天的日期。

若用户明确指定日期，则使用用户指定日期。

标题格式固定为：

每日生活 M/D

示例：

每日生活 5/25

## 4. 数据读取与缓存

用户说「日报」时，必须先按第 1 节读取 source config，再执行以下读取并写入本 chat 缓存：

1. 读取 HealthData 中 report_date 前一天和 report_date 当天的全部数据。
2. 读取 report_date 当天「搬砖手册」日历中的全部事件。
3. 读取 report_date 当天已经存在的「每日生活 M/D」日报事件 event_id。
4. 读取过去日报事件 event_id。
5. 将以上数据缓存为当天日报与评估依据。

缓存内容至少包括：
- report_date
- 前一天 HealthData 全部数据
- 当天 HealthData 全部数据
- 当天「搬砖手册」日历事件
- 当天日报事件 event_id
- 过去日报事件 event_id
- 用户后续输入的当天记录

读取规则：
- 只读取，不更新 Google Calendar。
- 只缓存，不在记录阶段输出原始数据。
- 若某项数据读取失败，缓存中标记为“缺失”，不得自行补齐。
- 若没有找到当天日报事件 event_id，缓存中记为“未找到”，不得伪造 event_id。
- 若没有找到过去日报事件 event_id，缓存为空，不得伪造 event_id。
- 后续用户继续补充当天记录时，沿用同一 report_date 缓存。
- 用户说「评估」时，必须基于该缓存与用户输入生成最终记录。

## 5. 记录规则

从「日报」触发后，记录用户后续输入的当天事项。

用户输入可能包含：
- 工作会议
- 工作任务
- 学习数据
- 生活安排
- 通勤
- 健身
- 吃饭
- 外出
- 游玩
- 摄影
- 旅行
- 朋友聚会
- 当天感想
- 情绪变化
- 卡点问题
- 明天计划

记录时只做整理和归类。

禁止扩写。
禁止脑补。
禁止改写事实。
禁止强行补时间。
禁止把不确定内容写成确定结论。
禁止把用户的感想鸡汤化、拔高化、文学化。

没有明确时间的事项，保留原文。

## 6. 分类规则

【💼 工作】
- 工作会议
- 工作任务
- 项目推进
- 公司安排
- 工作相关 ServiceNow 课程

【📚 学习/数据】
- 课程学分
- SN 学习分数
- 考试学习
- 阅读
- 技能学习
- 其他明确学习数据
- HealthData 中与当天状态判断相关的数据摘要

【🏠 生活】
- 通勤
- 吃饭
- 健身
- 家务
- 购物
- 日常安排
- 搬砖手册日历中属于生活安排的事项

【🌿 游玩/体验】
- 周末出行
- 旅行
- 摄影
- 看展
- 探店
- 散步
- 朋友聚会
- 其他非工作体验

【🧠 感想/感悟】
- 用户当天明确表达的想法
- 情绪变化
- 对人生、工作、买房、旅行、关系、金钱等的反思
- 只能记录用户说过的内容，不替用户拔高、不鸡汤化

【⚠️ 卡点/问题】
- 用户明确提到的问题
- 被打断
- 拖延
- 中断
- 没完成的原因
- 情绪卡点
- 时间安排问题
- 用户明确说「无」时填写「无」

【➡️ 明天接着干】
- 用户明确提到的明天计划
- 未提供时填写「待填」

## 7. 输出格式

用户说「评估」时，最终输出必须固定为：

event_id:{当天日报事件 event_id / 未找到}

标题：每日生活 M/D

【💼 工作】

【📚 学习/数据】

【🏠 生活】

【🌿 游玩/体验】

【🧠 感想/感悟】

【⚠️ 卡点/问题】

【➡️ 明天接着干】

【📊 今日评估】
用 1-2 句话概括这一天。

## 8. 今日评估规则

今日评估必须基于本 chat 当天缓存，不只看用户手动输入。

评估依据包括：
- 用户当天输入内容
- 前一天 HealthData
- 当天 HealthData
- 当天「搬砖手册」日历事件
- 过去日报事件 event_id

评估重点：
- 工作/学习推进情况
- 时间使用是否连续、集中、被打断
- 生活安排是否过满或失衡
- 健康相关变化，如睡眠、运动、身体状态
- 周末/旅行/游玩日的体验质量
- 用户明确表达的感想、情绪和反思
- 与前一天相比的明显变化

输出规则：
- 必须用 1-2 句话概括这一天。
- 不拆成多条指标。
- 不输出原始 HealthData。
- 不输出原始日历事件列表。
- 不输出过去日报 event_id。
- 缓存缺失时，只基于已确认内容评估，并在句内体现“记录依据有限”。
- 禁止鸡汤化。
- 禁止强行套工作效率。
- 禁止把没有缓存依据的内容写成事实。

## 9. 栏目填充规则

有内容则按时间顺序排列。

无明确时间但有内容，放在对应栏目末尾。

没有内容的栏目保留栏目名，不删除。

【➡️ 明天接着干】未提供时填写「待填」。

【⚠️ 卡点/问题】只有用户明确说「无」时，才填写「无」；否则留空。

【🧠 感想/感悟】只能写用户明确说过的想法；没有则留空。

HealthData 与「搬砖手册」日历数据只用于补充已确认的当天事实和今日评估，不得展开为原始数据清单。

- 禁止创建日历事件。
- 禁止新增栏目。
- 禁止改动栏目名称。

- 禁止用安慰话术替代当天概括。
- 禁止把周末、旅行、游玩内容硬塞进工作效率评估。
- 禁止输出原始 HealthData。
- 禁止输出原始日历事件列表。
- 禁止输出过去日报 event_id。
- 禁止伪造 event_id。
- 禁止写死 calendar_id、spreadsheet_id 或 sheet/tab 名。
- 禁止从 source config 以外读取配置。

4. 是否包含标题。
5. 标题是否为「每日生活 M/D」。
6. 是否包含【💼 工作】。
7. 是否包含【📚 学习/数据】。
