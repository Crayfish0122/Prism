# 😴 Somnilytics · 睡眠记录与分析系统

> 一句「起床了」换一份结构化睡眠反馈，数据始终待在你自己的 Google Calendar 里。

---

## 这是什么

一个配合 Google Calendar 使用的睡眠数据助手。它会在你起床时读取最近 7 天的睡眠事件作为个人基线，通过 5 个主观问题收集当天体感，然后输出一份固定五行格式的睡眠反馈——评级、证据、偏离项、结论、今晚策略。

助手只做纵向分析，不跨领域发散，不给多条建议，不编造阈值，也不更新日历。

---

## 你需要准备的东西

### Google Calendar（1 个日历）

新建一个日历，命名为「**灵魂返厂**」。每晚睡眠记录为一条事件，事件格式**必须严格遵守**如下约定，否则会被判为"解析失败"跳过：

**title** 固定为 `睡眠`

**description** 三行固定结构：

```
2026-04-16 04:28-13:30

inbed=542|core=206|deep=49|rem=127|
actual=382|debt_delta=38|status=轻度不足
```

三行分别是：日期 + 入睡-起床时段、在床与结构字段（分钟）、实际睡眠与负债状态。写入这部分目前由你自己或上游流程负责，Somnilytics 只负责读和分析。

### Project 挂载

把 `Prompt.txt` 放进 Claude Project 的 system instructions，把 `sleep_rules.md` 作为 source 文件挂在同一个 Project 下。两者有明确的优先级：冲突时以 `Prompt.txt` 为准。

> 时区与字段单位随 Google Calendar 本地设置走，建议保持一致避免跨时区日期漂移。

---

## 四个核心功能

### 1｜定调（读取与解析）

每天起床说一句「**起床了**」，助手会：

1. 从「灵魂返厂」日历按 `event start_time` 倒序读取最近 7 条有效睡眠事件
2. 按 `sleep_rules.md` 第 3、5 节解析原始字段并生成派生指标（awake / efficiency / deep_ratio / rem_ratio / core_ratio）
3. 将所有字段写入以"最新一天日期"为键的当日缓存，跨天清空

非睡眠事件、格式异常事件会被跳过；同日重复事件只保留排序最靠前那条；不足 7 条时按现有数量继续分析，并在证据行补一句"近 7 天基线不足，本次判断可信度下降"。

### 2｜补问（5 项主观字段）

定调完成后，助手**只针对最新一天**依次追问 5 个主观问题，一次一句，等你回完再问下一个。数字或关键词回答即可。

| # | 问题 | 接受输入 | 可读映射 |
|---|---|---|---|
| 1 | 恢复感怎么样？🌙 | 1–5 | 很差 / 偏差 / 一般 / 较好 / 很好 |
| 2 | 精神度呢？☀️ | 1–5 | 很差 / 偏差 / 一般 / 较好 / 很好 |
| 3 | 自然醒还是闹钟？⏰ | 自然醒 / 闹钟 | 自然醒 / 闹钟叫醒 |
| 4 | 夜醒几次？🌘 | 0 / 1 / 2 / 3+ | 无夜醒 / 夜醒 1 次 / … |
| 5 | 入睡前脑子乱还是平静？🧠 | 乱 / 平静 | 入睡前脑子乱 / 入睡前脑子平静 |

补问阶段不会展示规则来源、解释或选项说明，保持轻量。最终展示统一转为可读描述，必要时保留括号分数。

### 3｜反馈（固定五行输出）

5 项收齐后，助手一次性输出固定结构，每一行承担独立功能、不换词复述前文：

```
主观记录
{日期}：恢复感{描述+分数}，精神度{描述+分数}，{起床方式}，{夜醒描述}，{入睡状态描述}。

睡眠｜评级={好/一般/差}｜类型={主问题类型}｜执行={已执行/部分执行/未执行/无法判断/无前日建议}
证据｜总睡=…；入睡=HH:mm；结构={结构字段}；主观=…
对比｜第一偏离=…；第二偏离=…；归类={正常波动/已脱离近期常态/明显好于近期常态}
结论｜今天{更像/属于}{主问题}，它当前主要{拖累/改善}{最直接结果}。
今晚｜最该注意的是…；唯一动作是…

event_id: …
```

**主问题类型**只能 6 选 1：短睡型、长躺低效型、结构波动型、节律漂移型、主客观背离型、修复型。

**结构字段**只允许 6 个取值：正常、深睡偏低、深睡偏高、REM 偏低、效率偏低、信息不足。

全局不输出原始数据表、不输出 7 天明细、不给多条建议、不写套话、不更新 Google Calendar。

### 4｜前日执行判断

"昨晚有没有按昨天建议做"**默认由系统自动判断**，不列入固定提问。

可直接观测类建议（如"今晚早点睡""别把可睡窗口再压短""把入睡时点拉回去"）直接根据今天客观结果自动判定为 已执行 / 部分执行 / 未执行。不可直接观测类建议（如"睡前少刷手机""控制咖啡因"）标记为"无法判断"，只有在确实影响结论时才允许追加一句补问。

## 触发词速查

| 动作 | 触发词示例 |
|---|---|
| 启动分析 | 起床了 |
| 主观补问 | 助手逐条追问，无需主动输入 |
| 更新规则 | 规则更新了 / 上传新版 `sleep_rules.md` / 开新 chat |

除此之外的追问、改写、重新分析都会直接命中当日缓存，不会重复读取规则文件。

---

## 设计原则

- 最新一天是唯一主分析对象，最近 7 天只作基线不单独展示
- 结论必须由"客观数据 + 主观输入 + 7 天基线"三者共同支撑，缺一不写
- 同一判断全文最多出现 2 次（一次结论、一次依据），换词复述一律删除
- 每句话都必须提供新信息——判断、依据、后果、动作四者至少占一
- 证据不足时只能写"更像""主要表现为"，禁止"不是 A 也不是 B 而是 C"这种强判断结构
- 缺失字段只用可确认部分生成判断，不为凑完整补写不确定结论
- 规则缓存在单个 chat 内只读一次，补问和输出阶段不再引用 `sleep_rules.md`

---

## 一些使用上的小提醒

当日缓存以"最新一天日期"为隔离键，同一日期内的补问、重分析都写入同一份缓存；日期变化时自动清空旧缓存建立新缓存。

如果要调整**分析逻辑本身**（主问题分类、结构字段阈值、主观分数映射），改 `sleep_rules.md`；如果要调整**流程控制**（触发词、提问节奏、强制原则、禁止项），改 `Prompt.txt`。改完后开新 chat 即可生效，也可以在当前 chat 里说一声"规则更新了"让它重读一次。

同一天如果想撤回分析重走一遍，最简单的做法是开新 chat——缓存会自然重建，不会把旧判断拖进来。
