# sleep_rules.md

## 1. 睡眠事件原始读取内容

每个事件先读取以下原始内容：

- event_id
- title
- description
- start_time
- end_time

---

## 2. 睡眠事件文本格式

title:

睡眠

description:

2026-04-16 04:28-13:30

inbed=542|core=206|deep=49|rem=127|
actual=382|debt_delta=38|status=轻度不足

---

## 3. 字段解析规则

从 title 和 description 中解析以下字段：

1. 日期：从 description 第一行提取，例如 2026-04-16
2. 入睡时刻：从第一行时间段左侧提取，例如 04:28
3. 起床时刻：从第一行时间段右侧提取，例如 13:30
4. inbed：从 `inbed=数值` 提取
5. core：从 `core=数值` 提取
6. deep：从 `deep=数值` 提取
7. rem：从 `rem=数值` 提取
8. actual：从 `actual=数值` 提取
9. debt_delta：从 `debt_delta=数值` 提取
10. status：从 `status=文本` 提取
11. event_id：直接使用事件本身的 event_id

---

## 4. 解析后的后台字段

- 日期
- 入睡时刻
- 起床时刻
- inbed
- core
- deep
- rem
- actual
- debt_delta
- status
- event_id

---

## 5. 派生指标

- awake = inbed - actual
- efficiency = actual / inbed
- deep_ratio = deep / actual
- rem_ratio = rem / actual
- core_ratio = core / actual

若客观字段之间出现明显冲突或不合理值，不硬套派生指标；只保留可确认部分，并在后台标记该指标不纳入判断。

---

## 6. 异常处理

- 若 title 不是“睡眠”，优先按 description 是否符合睡眠格式判断是否纳入。
- 若最近7个事件中存在非睡眠事件，跳过，继续向前读取，直到凑满7个有效睡眠事件。
- 若 description 缺少某字段，则该字段记为“缺失”。
- 若 description 格式异常无法解析，则该事件保留原始文本入缓存，并标记“解析失败”。
- 若客观字段之间出现明显冲突或不合理值，不要硬套派生指标；只保留可确认部分。
- 后台校验说明只有在明显影响结论时，才可在正文中用一句话简短说明；否则不要写进正文。

---

## 7. 同日重复事件处理

若解析后发现同一日期存在多个有效睡眠事件：

- 只保留按排序规则最靠前的一条。
- 其余忽略。
- 忽略事件不纳入7天基线。

---

## 8. 有效事件不足处理

若最终可获得的有效睡眠事件少于7条：

- 按现有条数继续分析。
- 在证据行末尾追加：
  近7天基线不足，本次判断可信度下降

---

## 9. 缓存规则

每次触发“起床了”后：

1. 先读取最近7个有效睡眠事件。
2. 将每个事件的原始内容写入当天缓存：
   - event_id
   - title
   - description
   - start_time
   - end_time
3. 再将解析后的字段写入当天缓存：
   - 日期
   - 入睡时刻
   - 起床时刻
   - inbed
   - core
   - deep
   - rem
   - actual
   - debt_delta
   - status
   - awake
   - efficiency
   - deep_ratio
   - rem_ratio
   - core_ratio
4. 最新一天的主观输入、前一天建议、执行判断、当天分析结果、当天决策，也一并写入当天缓存。
5. 缓存按分析日期隔离，跨天清空，禁止混杂。

---

## 10. 缓存日期定义

- 当天缓存的隔离键，统一使用“最新一天”解析出的日期字段。
- 同一日期内的补问、补写、重新分析，都写入同一个当日缓存。
- 一旦最新一天日期变化，必须清空旧缓存并建立新缓存。

---

## 11. 主观字段

以下5项只针对“最新一天”补齐，并且最终必须展示为可读描述：

- 恢复感（1-5）
- 精神度（1-5）
- 起床方式（自然醒/闹钟）
- 夜醒（0/1/2/3+）
- 入睡状态（乱/平静）

---

## 12. 主观字段转换规则

恢复感：

- 1=很差
- 2=偏差
- 3=一般
- 4=较好
- 5=很好

精神度：

- 1=很差
- 2=偏差
- 3=一般
- 4=较好
- 5=很好

起床方式：

- 自然醒=自然醒
- 闹钟=闹钟叫醒

夜醒：

- 0=无夜醒
- 1=夜醒1次
- 2=夜醒2次
- 3+=夜醒3次及以上

入睡状态：

- 乱=入睡前脑子乱
- 平静=入睡前脑子平静

---

## 13. 分析原则

- 今天这晚睡眠是主分析对象。
- 最近7天只作为比较基线，不单独展示。
- 结论必须由“最新一天客观数据 + 最新一天主观输入 + 近7天基线”共同支撑。
- 不允许用重复同义句反复表达同一个判断。
- 不允许为了显得专业而强行贴标签。
- 没有证据支持时，不写“最明显异常”“结构不稳”“明显偏离”等重判断。
- 每一句都必须提供新信息：判断、依据、后果、动作四者至少有一项。
- 不做空泛安慰。
- 不做横向发散。
- 不写套话。

---

## 14. 主问题判定约束

每次必须只选一个最主要类型，不允许多选。

若证据不够强，只能写：

- 更像
- 主要表现为

禁止写：

- 不是A，也不是B，而是C

“主问题类型”只负责归类，不负责重复其他栏目已经说过的现象句。

---

## 15. 前一天决策执行反馈

### 15.1 判断昨天建议是否存在

若不存在：

- 执行情况 = 无前日建议

### 15.2 判断建议是否可观测

可观测建议示例：

- 别把可睡窗口再压短
- 今晚早点睡
- 把入睡时点拉回去
- 把整晚可睡时长留够
- 不要再继续缩短实际睡眠
- 先把总量守住

这类建议优先根据今天客观结果自动判断，不要问用户。

不可直接观测建议示例：

- 睡前少刷手机
- 主动放松
- 控制咖啡因
- 做睡前仪式

这类建议只有在确实影响判断时，才允许追加补问。

### 15.3 自动判断规则

- 已执行：今天出现了与昨天建议直接对应的可观测变化。
- 部分执行：出现了部分对应变化，但力度不够完整。
- 未执行：今天未出现对应变化，或相关指标继续朝反方向走。
- 无法判断：昨天建议不可直接观测，且没有足够证据支持自动判定。
- 无前日建议：昨天没有可用建议。

### 15.4 结果反馈规则

- 这里只评价“是否出现了与昨天建议直接对应的变化”。
- 不重复诊断今天整体睡得怎样。
- 若未出现与昨天建议直接对应的变化，写：
  今天未看到明确对应改善
- 若为“无法判断”，写：
  昨天建议属于不可直接观测行为，今天无法仅凭睡眠结果确认是否执行

---

## 16. 栏目分工约束

### 16.1 主观记录

- 只做主观事实记录。
- 只展示5项可读描述。
- 不做判断。
- 不埋结论。
- 不提前解释原因。

### 16.2 睡眠短反馈

主体输出压缩为5行：

1. 评级
2. 证据
3. 对比
4. 结论
5. 今晚策略

每一行必须承担独立功能，不得换说法重复前文。

### 16.3 评级行

只负责给出：

- 今天整体评级
- 主问题类型
- 前一天执行情况

不展开原因，不重复主观记录。

### 16.4 证据行

只负责给出最关键的客观与主观依据。

要求：

- 只写最关键的2-4个短分句。
- 不展开成长文。
- 若基线不足，只在这一行补一句：
  近7天基线不足，本次判断可信度下降
- 若前一天建议不可直接观测且无法自动判断，也只在这一行补一句：
  昨天建议不可直接观测，今天无法仅凭结果确认是否执行

### 16.5 对比行

只负责写：

- 相对近7天最明显的2个客观偏离项
- 整体归类

只能写客观偏离，不写解释，不写主观感受，不做结论复述。

归类只允许：

- 正常波动
- 已脱离近期常态
- 明显好于近期常态

### 16.6 结论行

只允许1句。

必须同时包含：

1. 今天更像哪一类
2. 它当前最直接影响了什么

不得展开铺垫，不得重复证据行。

### 16.7 今晚策略行

只保留：

- 1个今晚最该注意的点
- 1个唯一动作

不得重复诊断今天睡得怎样，不得给多条建议。

---

## 17. 结构字段生成规则

结构字段只允许使用以下值：

- 正常
- 深睡偏低
- 深睡偏高
- REM偏低
- 效率偏低
- 信息不足

优先级：

1. 如果关键字段缺失，写“信息不足”。
2. 如果 efficiency 明显低于近7天基线，写“效率偏低”。
3. 如果 deep_ratio 明显低于近7天基线，写“深睡偏低”。
4. 如果 deep_ratio 明显高于近7天基线，写“深睡偏高”。
5. 如果 rem_ratio 明显低于近7天基线，写“REM偏低”。
6. 若无明显异常，写“正常”。

---

## 18. 补充生成细则

- “证据”行最多写4个短分句，用分号分隔。
- “证据”行不得展开解释。
- “主观”字段只允许引用与判断最相关的1项或2项。
- 不得把5项主观记录再抄一遍。
- “结论”行只允许1句。
- “结论”行不得换词重复“评级行”和“对比行”的内容。
- “今晚”行只允许1个风险点 + 1个动作。
- 若部分客观字段缺失，仅使用可确认部分生成判断。
- 不得为了凑完整而补写不确定结论。
- 若前一天建议为不可直接观测类型且无法自动判断：
  - 首行执行=无法判断
  - 证据行末尾补：
    昨天建议不可直接观测，今天无法仅凭结果确认是否执行
- 若后台校验发现客观字段冲突或派生指标不可靠：
  - 仅用可确认字段判断
  - 只有在明显影响结论时，才能在证据行用一句短句提示：
    部分客观字段异常，本次判断基于可确认部分

---

## 19. 禁止事项全集

- 不展示客观原始数据表。
- 不展示最近7天客观表。
- 不做横向变量分析。
- 不用空洞词堆砌分析。
- 不写套话。
- 不给多条建议。
- 不更新 Google Calendar。
- 禁止用不同说法重复同一个判断。
- 禁止输出低信息总结句。
- 禁止把后台校验说明写成正文主体。
