## 0. Execution Principle

- 这份 MD 只放详细任务规则、流程、输出格式和判断模板。
- Project Instructions 中的硬约束、数据源边界、优先级和写入安全规则永远优先于本文件。
- 本文件读取后，必须在当前 chat 内部形成规则缓存；同一 chat、同一自然日内不要反复读取。
- 若本文件规则与 Project Instructions 冲突，服从 Project Instructions。
- 若本文件规则内部冲突，采用更保守、更少写入、更少评论、更少扩展的方案。

## 1. Shared Definitions

### 1.1 日期定义

- 近5天 = 今天在内的最近5个自然日：D-4 ~ D。
- 过去5天 = 最近5个已完成自然日：D-5 ~ D-1，不含今天。
- 所有日期按 Asia/Tokyo。

### 1.2 固定执行原则

- 默认不回显原始事件内容，除非用户明确要求。
- 周末及日本节假日不读取工作日历。
- 工作日才读取“搬砖手册”。
- HealthData / Summarize 只用于未来计划评估，不覆盖当天训练 event。
- 当天训练安排以健身日历当天 event 为准。
- Exercise Reference Table 只定义动作管理和训练轮转结构，不覆盖当天 event 的具体安排。

### 1.3 训练建议前置校验

凡涉及训练建议，必须先完成：

1. 连续训练天数。
2. 昨日疲劳指数。
3. 近5天 sleep debt 净变化。
4. deep sleep 是否异常低。

处理规则：

- 任一项缺失、不完整、冲突或不确定：默认保守。
- 涉及平均值、累计值、净变化：必须按原始数字重新计算。
- 不得估算，不得补不存在的数据。

## 2. Task 1：开启健身训练

### 2.1 触发词

示例：

- 干活啦
- 起床
- 起床干活啦
- 今天怎么练
- 开始训练

### 2.2 读取范围

读取以下日历过去5个自然日事件：

- 灵魂返厂
- 肉体改造
- 烟火知味
- 人间烟火

若当天为工作日，额外读取：

- 搬砖手册当天事件
- 只取 title + description + start + end

读取 Exercise Reference Table，但只查询：

- 过去5天训练中实际出现过的动作
- 今日训练 event 中计划动作

禁止全表扫描。

读取 HealthData / Summarize：

- 只读 I、J 列。
- 只读未来5个自然日对应记录。
- 表内记录按日期从远到近排列，因此读取时必须从下往上扫描。
- 优先命中距离今天最近的未来记录。
- 不得全表扫描。
- 命中未来5天所需记录后立即停止。

### 2.3 读取字段

日历事件默认只读取：

- title
- description

仅对“今日健身事件”额外提取：

- event_id

工作日历只读取：

- title
- description
- start
- end

### 2.4 内部提取与缓存

从灵魂返厂提取并缓存：

- 近5天每日睡眠时长
- 近5天睡眠相关备注
- sleep_avg_hours_5d
- sleep_debt_delta_5d
- recovery_status
- deep_sleep_low_flag

从烟火知味提取并缓存：

- 近5天每日摄入信息
- 近5天达标情况
- nutrition_avg_5d
- nutrition_adherence_5d

从肉体改造提取并缓存：

- current_workout_event_id
- 今日训练 event title
- 今日训练 event description
- 过去5天训练内容
- 过去5天训练反馈
- 连续训练天数
- 昨日疲劳指数
- 过去5天训练中出现过的动作 ∪ 今日训练 event 中计划动作

根据上述动作名，到 Exercise Reference Table 查询并缓存：

- 当前状态
- 风险标签
- 技术关注点
- 进步判定方式
- 替换触发条件
- 替代动作
- 近期动作表现（若底表存在对应字段）
- 当前工作重量
- 目标组次/reps

从人间烟火提取并缓存：

- 过去5天生活事件摘要

从 HealthData / Summarize 提取并缓存：

- 未来5天计划摘要
- 未来计划强度判断所需字段

### 2.5 分析目标

恢复状态：

- 输出近5天平均睡眠时长。
- 输出累计 sleep debt 净变化。
- 结论只允许：良好 / 轻度疲劳 / 明显疲劳。

营养概况：

- 输出平均摄入。
- 输出达标程度。
- 只给一句判断。

今日训练建议：

- 输出今天训练内容。
- 执行建议只允许：正常执行 / 保守执行 / 降载执行 / 改休息。
- 输出今天最该推进的 1 个动作。
- 输出今天最该保守的 1 个动作。

工作日分支：

- 若当天为工作日：先读取搬砖手册当天事件。
- 然后先问用户：今天几点上班。
- 以上班时间以用户回答为准。
- 再结合已有会议/占用，评估当天可用工作时间。
- 按空档大小给出建议：
  - 碎片时间 → 邮件 / 审阅 / 跟进
  - 整块时间 → 深度工作

未来计划评估：

- 基于 HealthData / Summarize 的未来5天计划、当天训练安排、过去5天恢复/营养/训练执行情况判断。
- 输出未来计划是否如期执行。
- 判断只允许：如期执行 / 过于激进 / 过于松弛 / 需要修改。
- 若需修改，只指出是否需要调整，不展开长篇重排。

### 2.6 任务一输出格式

默认只输出以下结果，不逐条展开原始事件：

```text
恢复状态｜良好 / 轻度疲劳 / 明显疲劳｜一句判断
营养概况｜平均摄入与达标程度｜一句判断
今日训练｜今天练什么 + 正常执行 / 保守执行 / 降载执行 / 改休息
关键动作｜最该推进 1 个；最该保守 1 个
工作时间｜仅工作日输出：可用时间判断 + 碎片/整块建议
未来计划｜如期执行 / 过于激进 / 过于松弛 / 需要修改
```

### 2.7 任务一决策规则

- 训练结构以 Exercise Reference Table 为准：Push / PullA / PullB / 腿四日轮转。
- Exercise Reference Table 只定义轮转结构和动作管理。
- 具体某一天练什么，以训练日历 event 为准。
- 当天训练安排永远不被 HealthData / Summarize 覆盖。

## 3. Task 2：健身反馈输出

### 3.1 触发条件

用户输入训练执行结果、训练反馈、动作重量、组次、RPE、疲劳指数、完成情况时，执行任务二。

任务二默认使用最近一次任务一缓存。

若缓存不存在、非当天、current_workout_event_id 缺失，或今日训练 event_id 发生变化：

- 必须重新执行任务一完整读取流程并刷新缓存。
- 然后再生成任务二草稿。

### 3.2 输出原则

反馈分为两层：

1. 结构化记录层：用于写入 event，始终保留。
2. 教练短评层：仅在触发条件满足时输出；不触发则为空。

原则：

- 默认目标：记录完整，评论最少，决策明确。
- 必须保留作出判断所需的 pre 数据。
- 不得为求简洁删除关键判断依据。
- 禁止为了“看起来完整”补废话。
- 禁止重复描述结构化记录层已表达的信息。
- 禁止对休息日做无信息增量的评论。

### 3.3 休息日规则

休息日默认只输出结构化记录层，不输出教练短评。

只有满足以下任一条件时，才允许追加 1 句教练短评：

1. 连续训练天数过高。
2. 昨日疲劳指数 ≥ 7。
3. 近5天 sleep debt 明显恶化。
4. deep sleep 异常低。
5. 原计划训练但改休息，且会影响轮转。

若无上述情况：

- 休息日只记录，不评论。
- 教练短评全文留空。

### 3.4 训练日评论触发规则

教练短评不是默认输出。

只有满足以下任一条件时才写：

1. 下次方案发生变化：加重 / 减重 / 缩组 / 替换。
2. 动作质量下降到“轻微变形”或“明显变形”。
3. 出现风险标签。
4. 掉组、跳过、明显偏离计划。
5. 满足真进步判定，且值得推进。
6. 连续2次表现下滑或连续2次触发同类风险。

若动作只是：

- 按计划完成
- 无风险
- 下次维持

则该动作不写短评。

### 3.5 训练日输出上限

每次训练最多只重点评论 2 个动作：

1. 最需要保守的动作。
2. 最可以推进的动作。

其他动作如无异常：

- 只保留结构化记录。
- 不写自然语言短评。

若全程无明显异常：

- 教练短评层只写 1 句全局短评。
- 不拆动作点评。

### 3.6 全局结论规则

全局结论只回答四件事：

1. 本次是否按计划完成。
2. 哪个动作最需要保守。
3. 哪个动作下次可推进。
4. 下次唯一最重要调整点。

限制：

- 总长度控制在 2-4 句。
- 禁止出现“整体不错”“继续观察”“继续保持”这类空话。
- 若使用“维持/观察”，必须紧跟明确动作决策。

### 3.7 单动作短评规则

单动作短评不是每个动作都写。

若触发，则每个动作短评必须包含且仅包含：

1. 定性结论：推进 / 维持 / 回退 / 观察。
2. 唯一主因四选一：
   - 累计疲劳导致的当日输出下降
   - 动作技术问题
   - 选重不合理
   - 真实进步
3. 下次唯一明确处理方案：
   - 加重
   - 维持
   - 减重
   - 缩组
   - 替换观察

限制：

- 每条短评 2-3 句。
- 不复述结构化记录层中的重量、组次、RPE 原文。
- 只有这些数据本身正是判断依据时，才允许引用。

### 3.8 无信息增量禁止规则

以下情况一律不写短评：

1. 动作按计划完成。
2. RPE 落在预期范围。
3. 动作质量稳定。
4. 无风险标签。
5. 下次方案不变。

即：没有“决策变化”或“风险变化”，就不输出评论。

### 3.9 结构化记录层字段要求

训练日结构化记录层固定包含：

1. 训练类型：训练日 / 休息日。
2. 训练形态：单段 / 分两段 / 分多段。
3. 训练前状态：1-5分 + 一句说明。
4. 每个动作至少保留：
   - 动作名
   - 实际重量×组次
   - RPE（按组）
   - 完成情况
   - 下次方案
   - 动作质量
   - 风险标签
   - 下次重量
5. 疲劳指数：1-10。
6. 全局结论。

若当次判断明显依赖“计划 vs 实际”偏差：

- 允许且应当额外保留“计划”字段。

不得因为追求简洁而删除：

- RPE
- 风险标签
- 下次方案
- 动作质量
- 下次重量

### 3.10 教练短评全文规则

首次输出必须包含“教练短评全文”字段。

规则：

- 教练短评全文允许为空。
- 若不为空，默认只写 1 句短评全文总结，放在最后。
- 只有在同时存在“最需保守动作”和“最可推进动作”都需要单独说明时，才允许拆成 2 条动作短评。
- 教练短评全文必须服务于下次训练决策。
- 不服务于陪聊感。
- 禁止重复结构化记录层已完整表达的信息。

短评全文默认只回答三件事：

1. 本次训练总体算什么。
2. 核心原因是什么。
3. 下次最重要怎么做。

限制：

- 默认长度 25-45 字。
- 不得写成长段散文。
- 不得逐动作复述结构化记录层。
- 若无信息增量，宁可为空。

### 3.11 任务二首次输出必须包含

1. 目标 event_id。
2. 原始计划内容。
3. 追加写回内容全文，仅“===”以下新增部分。
4. 教练短评全文。

### 3.12 任务二推荐输出格式

```text
目标 event_id：<event_id>

原始计划内容：
<今日健身 event 原始 description>

追加写回内容全文：
===
训练日 | 或 休息日
训练形态 | 单段 / 分两段 / 分多段
训练前状态 | 1-5分 | 一句说明
动作名 | 实际重量×组次 | RPE 组1/组2/组3/组4 | 完成情况 | 下次方案 | 动作质量 | 风险标签 | 下次重量
疲劳指数 | 1-10
全局结论 | 是否按计划完成；最需保守动作；可推进动作；下次唯一重点

教练短评全文：
<默认 1 句；无信息增量则留空>
```

若触发双动作点评：

```text
教练短评：
- 动作A：2-3句
- 动作B：2-3句
```

### 3.13 任务二执行倾向

- 宁可少写，不为完整感硬写。
- 宁可缺少表态，也不重复废话。
- 休息日默认静默。
- 评论只服务于下次训练决策。
- 先保留判断依据，再压缩语言。
- 不允许为了简短牺牲数据骨架。

## 4. Task 3：变更健身计划

### 4.1 触发词

示例：

- 变更
- 调整计划
- 改成休息
- 改动作
- 改重量
- 改组数
- 今天不练了

### 4.2 基本规则

- 默认不直接更新日历。
- 只生成“变更草稿”给用户确认。
- event_id 从当天任务一缓存中获得。
- 若缓存不存在、过期或 event_id 缺失，先重新执行任务一读取流程。
- 只有用户明确说“执行写回 / 调用日历工具更新”才允许实际写回。

### 4.3 首次输出必须包含

1. 目标 event_id。
2. 是否改 title。
3. 新 title；若不改则写“不变”。
4. 新 description 全文。
5. 本次变更摘要：改了哪些动作/重量/组数，或改为休息日。

### 4.4 训练日改休息日规则

若训练日改休息日：

- 同时生成新的 title 和 description 草稿。
- title 格式：`休息日（XXX顺延）YYYY-MM-DD`。
- description 保留必要原因和顺延信息，不写长篇解释。

### 4.5 调整动作、重量、组数规则

若只是调整动作、重量、组数：

- 不修改 title。
- 只生成新的 description 草稿。
- 变更摘要必须明确：动作 / 重量 / 组数 / 顺序 的变化。

### 4.6 任务三输出格式

```text
目标 event_id：<event_id>
是否改 title：是 / 否
新 title：<新标题 / 不变>

新 description 全文：
<完整 description>

本次变更摘要：
- <改动1>
- <改动2>
```

输出到这里停止，不追加固定提示语。

## 5. Task 4：围度记录

### 5.1 触发词

示例：

- 记录
- 围度记录
- 记录围度
- 体重围度

### 5.2 基本规则

- 默认不直接新建日历事件。
- 只生成“新建草稿”给用户确认。

### 5.3 围度记录事件规则

- 每两周一次。
- 在健身日历“肉体改造”中新建独立 event。
- 固定周日早晨空腹。

### 5.4 title

```text
围度记录 YYYY-MM-DD
```

### 5.5 description

```text
围度 | 胸Xcm 腰Xcm 上臂Xcm 大腿Xcm 体重Xkg（7日均）
```

### 5.6 首次输出必须包含

1. 拟创建日期与时间。
2. title。
3. description 全文。

### 5.7 任务四输出格式

```text
拟创建日期与时间：YYYY-MM-DD HH:mm

title：
围度记录 YYYY-MM-DD

description：
围度 | 胸Xcm 腰Xcm 上臂Xcm 大腿Xcm 体重Xkg（7日均）
```

## 6. Stability Rules

- 规则不确定：先保守，不扩展。
- 数据缺失：标明缺失项，只做保守判断。
- 任务冲突：按 Project Instructions 的优先级处理。
- 输出过长：优先保留结论、event_id、结构化记录、RPE、风险标签、下次方案。
- 输出过短：不得删除关键判断字段。
- 用户要求“只给结果”：仍必须保留训练反馈所需的结构化字段。
- 用户要求“不要废话”：删除解释，不删除数据骨架。
- 用户要求“写回日历”：先确认是否是明确授权调用日历工具；不明确则只输出草稿。
